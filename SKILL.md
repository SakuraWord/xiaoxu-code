# 开发工作流

## 概述

遵循结构化的开发流程，确保每一步都经过思考和确认。核心原则：**方案先行，审批后动工，遇疑必问，完成必验。**

## 流程总览

```
需求分析 → 技术方案 → 方案审批 → 开发准备 → 编码实现 → Code Review → 测试
```

**硬门槛：方案审批**——你没有说"可以"/"开始"/"没问题"之前，绝不动手写代码。

---

## 第一步：需求分析

搞清楚功能要达成什么效果。结合当前项目的实际代码来理解，而不是凭空猜测。

1. 阅读相关代码文件，理解现有架构和模式
2. 明确功能边界：做什么、不做什么
3. 识别影响范围：哪些文件/模块会被改动
4. 如果不确定需求的任何细节，向用户提问澄清

完成后，用一两句话总结你理解的需求，等待用户确认后再进入下一步。

---

## 第二步：技术方案

调用 `superpowers:brainstorming` 技能来设计方案。方案需要覆盖以下内容，不得遗漏：

- **改动文件清单**：精确到文件路径，标注是新建还是修改
- **数据结构设计**：涉及到的 model/interface/type/schema 变更
- **组件/模块拆分**：如果涉及多个模块，说明各自的职责和边界
- **API/接口定义**：函数签名、API 端点、参数和返回值
- **数据流**：数据如何在模块间流转

方案中引用的文件名、函数名、API 必须是代码中真实存在的——读代码确认过，不是凭空编的。

## 第三步：方案验证与审批

1. 检查方案的可行性：是否与现有架构兼容、是否有遗漏的边界情况
2. 整理方案并提交给用户
3. **等待用户明确批准**——用户说"可以"、"没问题"、"开始吧"等才继续

**用户没有明确批准之前，禁止写任何代码、创建任何文件。**

## 第四步：开发准备

询问用户是否需要创建新分支：

- **要创建**：请用户输入分支名称，然后创建并切换到该分支
- **不创建**：在当前分支继续开发

如有需要，运行项目依赖安装和基础测试验证环境正常。

## 第五步：编码实现（TDD 驱动）

使用测试驱动开发的方式实施改动。调用 `superpowers:test-driven-development` 技能来指导编码过程。

TDD 的核心铁律：**先写测试，看它失败，再写最简代码让它通过。** 没有测试的代码不许写。

具体方式：
- 按照技术方案中列出的文件和模块顺序，逐个实现
- 每个功能单元：先写测试 → 确认测试失败 → 写最小实现代码 → 确认测试通过
- 测试用例必须使用真实代码，避免过度 mock

**遇到以下情况必须停下并向用户提问：**

- 方案中没有覆盖的实现细节
- 多种可能的实现方式，不确定选哪个
- 改动的副作用不明确
- API 或数据结构需要在实施中调整
- 发现方案中的问题或遗漏

不要猜测，不要假设。不确定就问。改动范围严格限定在方案中列出的文件——不做未经审批的额外"改进"。

## 第六步：Code Review

代码写完后，调用 `superpowers:requesting-code-review` 技能，派发独立的 code reviewer 子代理对改动进行系统性审查。

审查必须覆盖：
- 逻辑正确性：代码逻辑是否有 bug
- 方案一致性：实现是否与技术方案一致
- 边界情况：是否遗漏了错误处理、空值、竞态等场景
- 代码规范：是否遵循项目的代码风格和模式

审查发现的问题必须修正后才能继续。修正后重新确认编译/类型检查通过。

## 第七步：集成验证

TDD 步骤已覆盖了单元测试，这一步根据改动量决定集成测试范围：

- **多文件改动（3 个及以上文件）**：测试相关模块的联动功能，确保没有破坏上下游。运行项目已有的端到端测试或集成测试。
- **少量改动（1-2 个文件）**：聚焦运行当前模块已有的测试，确保回归通过

步骤：
1. 运行项目全部测试，确认全部通过
2. 如果有端到端测试或集成测试，务必运行一遍
3. 手动验证关键路径的功能是否符合预期

## 第八步：完成确认

汇总本次开发的内容：
- 改动了哪些文件
- 实现了什么功能
- 测试结果
- 是否有遗留问题或后续需要关注的点

如有分支合并操作，**合并后绝对不能删除功能分支**，保留所有分支以便回溯和后续维护。

---

## 关键原则

- **遇疑必问**：任何不确定的事，停下来问用户，不要自己猜
- **方案先行**：先想清楚再动手，不凭直觉写代码
- **最小改动**：只改方案里列出的东西，不做额外"顺手"的修改
- **真实代码**：方案必须基于实际代码，不是凭空编造
- **审批硬门槛**：方案未经用户确认，禁止写代码
- **合并留分支**：合并后不得删除功能分支，保留完整的开发历史
